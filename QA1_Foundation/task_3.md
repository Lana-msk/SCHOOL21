# Сравнение методологий разработки ПО


| Методология | Плюсы | Минусы / Риски | Пример проекта / типа компании |
|-------------|-------|----------------|--------------------------------|
| **Waterfall** | • Четкая документация<br>• Простота управления<br>• Предсказуемые сроки и бюджет | • Сложность внесения изменений<br>• Позднее тестирование<br>• Риск несоответствия требованиям | Крупные государственные проекты (например, система управления воздушным движением) |
| **V-Model** | • Раннее тестирование<br>• Высокое качество продукта<br>• Четкая связь этапов | • Негибкость к изменениям<br>• Высокие затраты на документацию<br>• Длительный цикл разработки | Медицинские системы, аэрокосмические проекты (например, ПО для МРТ-аппаратов) |
| **Incremental** | • Быстрое получение частичного результата<br>• Гибкость в планировании<br>• Снижение рисков | • Сложность интеграции модулей<br>• Требует тщательного планирования<br>• Возможна несогласованность компонентов | Крупные банковские системы (например, поэтапное обновление интернет-банка) |
| **Scrum** | • Быстрая адаптация к изменениям<br>• Регулярная обратная связь<br>• Высокая мотивация команды | • Требует опытного Scrum-мастера<br>• Сложность оценки сроков<br>• Риск "спринтной усталости" | Стартапы, веб-разработка (например, мобильное приложение для доставки еды) |
| **Kanban** | • Гибкость процессов<br>• Прозрачность процессов<br>• Непрерывная поставка | • Отсутствие временных рамок<br>• Сложность планирования<br>• Риск накопления незавершенных задач | IT-поддержка, сервисные компании (например, команда технической поддержки SaaS-продукта) |

# Agile-манифест/идеи


| Идея манифеста | Что это значит на практике? |
|----------------|-----------------------------|
| **Люди и взаимодействие важнее процессов и инструментов** | 1. Ежедневные стендапы для живого общения вместо формальных отчетов<br>2. Совместные рабочие сессии для обмена знаниями |
| **Работающий продукт важнее исчерпывающей документации** | 1. Демонстрация работающего функционала на каждом спринте<br>2. Фокус на создании минимально работоспособного продукта (MVP) |
| **Сотрудничество с заказчиком важнее согласования условий контракта** | 1. Регулярные встречи с продукт-оунером для уточнения требований<br>2. Включение представителя заказчика в команду разработки |
| **Готовность к изменениям важнее следования первоначальному плану** | 1. Регулярный ретроспективный анализ и адаптация процессов<br>2. Гибкое перепланирование спринтов на основе обратной связи |

* Комментарий к фразе манифеста

Фраза «То есть не отрицая важности того, что справа, мы всё-таки больше ценим то, что слева» означает, 
что Agile-манифест не призывает полностью отказаться от процессов, документации, контрактов и планов. 
Вместо этого он предлагает изменить приоритеты: использовать процессы и инструменты как средство для эффективного взаимодействия, 
создавать документацию в необходимом объеме для поддержки работающего продукта, соблюдать условия контрактов через активное сотрудничество, 
и следовать планам, оставаясь открытыми к изменениям. Это философия баланса, где традиционные ценности не отвергаются, но переосмысливаются в контексте гибкой разработки.
